### Open Food Facts (optional offline enrichment only)

**Not used at runtime.** Optional step in `scripts/build-demo-categories.mjs`:

```bash
OPEN_FOOD_FACTS=1 OFF_BUDGET=200 npm run build:demo-categories
```

Flow: classify via local knowledge → for remaining barcode unknowns, `GET world.openfoodfacts.org/api/v2/product/{barcode}` → map `categories_tags_en` → EasiShop aisle slug → write static JSON. Rate-limited (`sleep` ~120ms) + budgeted.

**Not sustainable as production category determination or live filtering:** weak coverage for SA private label / beauty / household; taxonomy mismatch; rate limits; crowd-sourced errors; doesn’t grow with scrapers. OK only as occasional bootstrap/gap-fill when building a first DB category column.

### Production categories — DB vs frontend filtering

**Cofounder plan (recommended):** add a category column (or categories FK) and rewrite scrapers to label Category on DB commit.

| Approach | Role |
|----------|------|
| **Frontend heuristics / offline map** | Demo bridge only — does not scale to ~99k, pagination, or multi-client consistency |
| **DB + scrapers at ingest** | Production source of truth for aisle browse, deals filters, PDP crumbs, API facets |

**Verdict:** a DB (and API) category field is **necessary** for real category filtering. Frontend-only tagging is fine as a temporary fallback when `category` is null, not as the long-term system.

Suggested hybrid: align slug/name with `CATEGORIES` in `catalog.ts` → scrapers write on insert/update → one-time backfill (retailer department → heuristics → optional OFF for barcode gaps) → expose on product API → drop demo map when live catalog carries aisles.
